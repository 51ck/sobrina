**Implementation note (T14.1):** `recordCheckIn(store, chatId, memberId, dayKey, intent)` in `packages/core/src/checkin.ts` implements `sober` only; `intent: "slip"` is accepted in the type surface (so T14.2 extends the same call sites) but throws until T15's `resolveSlip` is wired in. Requires the member already on the Checklist — throws `NotOnChecklistError` rather than auto-joining, leaving the join+record composition to T14.3. Uses `ensureOpenDay` (not a plain read) so an early Check-in still opens its own Day; a `closed` Day throws `DayClosedError` (early slice of T14.4's full policy). Calls `maybeEarnGraceToken(..., currentStreak = 0, graceTokenN)` per T15's interim note — grants nothing yet since `graceTokenN >= 1`; T14.2/T18.1 should pass the real Streak count once available. See module doc in `checkin.ts` for the full rationale.

**Implementation note (T14.2):** `recordCheckIn(..., "slip")` now calls `resolveSlip(store, chatId, memberId)` (T15.2) inside the same open-Day + Checklist-membership transaction as the sober path, then UPSERTs `check_ins` with the returned `status` (`minor_slip`/`major_slip`) and `spent_grace_token`. `resolveSlip`'s own transaction nests via SQLite `SAVEPOINT` (bun:sqlite `Database.transaction` supports nesting) — verified by the passing test suite. `maybeEarnGraceToken` is never called for `slip` (explicit slip is not sober progress). Same-Day overwrite resolves fresh every call — no caching of "already spent this Day" — so re-slipping after the first slip already spent the token correctly yields `major_slip` (not a double-spend) but can *upgrade* a `minor_slip` to `major_slip` on a same-Day re-slip; documented in `checkin.ts` module doc and covered by a test. Sober-over-slip overwrite refunds a spent token via `refundGraceToken` before clearing `spent_grace_token` (T14.1's deferred note, now implemented — cheap given the row is already read to decide the refund); the inverse (slip-over-slip re-clearing) does not restore a token since it was already cleared, which is a documented asymmetry for T17/T18 to be aware of. Reviewed by cross-model Standards + Spec subagents (both approved).

**Implementation note (T14.3):** `joinAndRecordCheckIn(store, chatId, memberId, dayKey, intent)` in `checkin.ts` — Option A, a thin composed verb: `joinChecklist` (T12.1, idempotent) then `recordCheckIn` unchanged (T14.1/T14.2), passed straight through for both `sober` and `slip`. `recordCheckIn` itself stays strict (`NotOnChecklistError` for direct non-member calls) — chosen over Option B (`{ joinIfNeeded }` flag) so no existing call site's membership requirement changes. `joinChecklist` and `recordCheckIn` each run as their own atomic statement/transaction (no single transaction spans both) — acceptable for a single in-process call. Already-member calls behave exactly like calling `recordCheckIn` directly (`joinChecklist` no-ops for an active member). Reviewed by cross-model Standards + Spec subagents (Standards flagged a missing `@sobri/core` barrel export in `index.ts`, fixed; both approved after fix).

**Implementation note (T14.4):** No new guard code — `DayClosedError` (T14.1/T14.2) and `NotOnChecklistError` (T14.1) already existed and already covered both intents and both verbs (`recordCheckIn` direct + `joinAndRecordCheckIn` via T14.3). This slice's deliverable was the **full reject-case policy documentation** (a "Reject cases" table added to `checkin.ts`'s module doc: Day closed → `DayClosedError`, not on Checklist → `NotOnChecklistError` (`recordCheckIn` only — `joinAndRecordCheckIn` joins first so it never hits this), blank `chatId`/`memberId`/`dayKey` → plain `Error`; no other reject cases exist in the module) plus closing test gaps: (1) `joinAndRecordCheckIn` with `slip` intent onto a closed Day (only `sober` had a test before — join still lands, write still rejects), and (2) two "no-mutation" tests that put a real Check-in row on the Day *before* closing it (one `sober`, one `minor_slip` with a spent token), close the Day, then attempt the **opposite** intent and assert the stored row's `status`, `spent_grace_token`, and `updated_at` are unchanged — this is the case most likely to leak a mutation, since the closed-Day throw must fire before `writeSober`'s refund logic (which reads the row, then updates the Grace Token store) or `writeSlip`'s `check_ins` UPSERT ever runs. Reviewed by cross-model Standards + Spec subagents (both approved).
